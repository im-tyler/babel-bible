## Intuition [Beginner]

Think about a notebook that holds 10 pages. You write entries one by one. Most of the time, you just write on the next blank page. But when the notebook fills up, you need to buy a new notebook with twice as many pages and copy all your old entries over. That copy takes a lot of time — much more than a single write.

Does this mean writing an entry is slow? Not really. The copy happens only once in a while. If you wrote 100 entries, the copy happened maybe once or twice. Averaged over all 100 writes, each one cost about as much as a single write plus a small fraction of the copy.

This is amortized analysis. Instead of asking "how much does the worst single operation cost?" it asks "how much do $n$ operations cost in total, per operation on average?" The worst-case cost of a single operation might be large (the notebook copy), but the amortized cost over a sequence is small.

There are three ways to compute the amortized cost. The *aggregate method* simply totals the work and divides. The *accounting method* (or token method) assigns a fixed charge to each operation, saves the surplus from cheap operations as credits, and spends credits on expensive ones. The *potential method* generalises the accounting method by defining a potential energy stored in the data structure that is released when expensive operations happen.

## Visual [Beginner]

A dynamic array growing by doubling:

```
Insert into array of capacity 4 (size 3):
[1][2][3][ ]  →  [1][2][3][4]  ← cheap: O(1)

Insert into array of capacity 4 (size 4, full):
[1][2][3][4]  →  [1][2][3][4][5][ ][ ][ ]  ← expensive: allocate + copy
                new capacity 8

Insert into array of capacity 8 (size 5-7):
[1][2][3][4][5][6][ ]  ← cheap: O(1) each

Insert into array of capacity 8 (size 8, full):
→ allocate 16, copy 8 items  ← expensive again
```

| Operation | Actual cost | Credits saved | Credits spent | Amortized cost |
|---|---|---|---|---|
| Insert (not full) | 1 | +2 | 0 | 3 |
| Insert (triggers resize) | $n + 1$ | 0 | $-n$ | 3 |
| Insert (not full) | 1 | +2 | 0 | 3 |

The accounting method charges 3 tokens per insert. Cheap inserts bank 2 tokens. The resize spends the banked tokens on copying.

## Worked example [Beginner]

A dynamic array starts with capacity 1 and doubles when full. Each insert into a non-full slot costs 1 (write the element). Each insert that triggers a resize costs $n + 1$ (copy $n$ old elements, write the new one, where $n$ is the current size before doubling).

Let us total the cost of $n$ insertions starting from an empty array. The resizes happen at sizes 1, 2, 4, 8, ..., up to the largest power of 2 below $n$. The total copy cost is $1 + 2 + 4 + \cdots + n/2 + n < 2n$ (a geometric series). The $n$ writes each cost 1. So the total cost is at most $3n$.

The amortized cost per insertion is $3n / n = 3$. Each insert costs $O(1)$ amortized, even though individual inserts can cost $O(n)$.

## Formal definition [Intermediate+]

Let a data structure undergo a sequence of $n$ operations with actual costs $c_1, c_2, \ldots, c_n$. The *total actual cost* is $T = \sum_{i=1}^{n} c_i$.

**Aggregate method.** Compute $T$ directly and report the amortized cost as $T / n$.

**Accounting method.** Assign an *amortized cost* $\hat{c}_i$ to each operation $i$. Require that the credit balance never goes negative: for all $k$, $\sum_{i=1}^{k} \hat{c}_i \ge \sum_{i=1}^{k} c_i$. Then $\sum_{i=1}^{n} \hat{c}_i \ge T$, so the amortized costs upper-bound the actual total cost.

**Potential method.** Let $\Phi$ be a *potential function* mapping the state $D_i$ of the data structure after operation $i$ to a non-negative real number, with $\Phi(D_0) = 0$ for the initial state. The *amortized cost* of operation $i$ is:

$$\hat{c}_i = c_i + \Phi(D_i) - \Phi(D_{i-1}).$$

The total amortized cost is:

$$\sum_{i=1}^{n} \hat{c}_i = \sum_{i=1}^{n} c_i + \Phi(D_n) - \Phi(D_0) \ge \sum_{i=1}^{n} c_i,$$

since $\Phi(D_n) \ge 0$ and $\Phi(D_0) = 0$.

### Counterexamples to common slips

- *"Amortized $O(1)$ means every operation is $O(1)$."* No. Individual operations can be expensive ($O(n)$). The amortized bound says the average over a sequence is $O(1)$, not that each operation is.

- *"The accounting method and potential method are different analyses."* The accounting method is a special case of the potential method where the potential equals the total credit balance. The potential method is more flexible because $\Phi$ can be any function of the data structure state.

- *"Amortized analysis works for worst-case inputs."* It does: the bound holds for any sequence of $n$ operations, not just for average-case or random inputs. The averaging is over operations in the sequence, not over probability distributions.

### Key theorem with proof [Intermediate+]

**Theorem (dynamic table: $O(1)$ amortized insertion).** *Consider a dynamic table that doubles its capacity when full. Starting from an empty table with capacity 1, the amortized cost of each insertion is $O(1)$.*

*Proof (potential method).* Define $\Phi(D_i) = 2 \cdot \text{size}(D_i) - \text{capacity}(D_i)$, where $\text{size}$ is the number of elements and $\text{capacity}$ is the allocated array size. Note $\Phi \ge 0$ always (size never exceeds capacity) and $\Phi(D_0) = 0$.

Case 1: Insertion without resize (size $< $ capacity before insertion). Actual cost $c_i = 1$. After insertion, size increases by 1:

$$\hat{c}_i = 1 + 2(\text{size} + 1) - \text{cap} - (2 \cdot \text{size} - \text{cap}) = 1 + 2 = 3.$$

Case 2: Insertion with resize (size $= $ capacity $= k$ before insertion). Actual cost $c_i = k + 1$ (copy $k$ elements, write new element, capacity becomes $2k$). After insertion, new size is $k + 1$, new capacity is $2k$:

$$\hat{c}_i = (k + 1) + 2(k + 1) - 2k - (2k - k) = k + 1 + 2 - k = 3.$$

In both cases $\hat{c}_i \le 3 = O(1)$. $\square$

**Bridge.** The potential method is the foundational reason amortized analysis works as a rigorous tool: the potential $\Phi$ captures the "stored work" in the data structure, and the amortized cost formula $\hat{c}_i = c_i + \Delta\Phi$ ensures that total amortized cost upper-bounds total actual cost. This builds toward the analysis of red-black trees in [47.04.02], where rotations during rebalancing have $O(1)$ amortized cost, and it appears again in [47.04.03] where augmenting paths in network flow are charged against a potential that decreases with each augmentation. The central insight is that expensive operations consume potential built up by cheap ones, so the potential function serves as a "prepayment" scheme, and putting these together gives tight bounds on data structure performance that worst-case analysis alone cannot achieve.

**Exercise 3 (medium, symbolic).**

A stack supports PUSH (cost 1) and MULTIPOP($k$) (cost $\min(k, s)$ where $s$ is the current stack size). Using the aggregate method, show that $n$ operations have total cost $O(n)$.

<details class="exercise__answer"><summary>Answer</summary>

Each element is pushed at most once and popped at most once (whether by a regular pop or by a multipop). Over $n$ operations, the total number of pushes is at most $n$, and the total number of individual pops across all multipops is at most $n$ (since you cannot pop more items than were ever pushed). So the total cost is at most $2n = O(n)$.

</details>

<aside class="exercise" data-difficulty="medium" data-type="symbolic" markdown="1">

**Exercise 4 (medium, symbolic).**

Use the accounting method to show that PUSH on a stack has $O(1)$ amortized cost and MULTIPOP has $O(1)$ amortized cost. Assign tokens.

<details class="exercise__answer"><summary>Answer</summary>

Charge PUSH an amortized cost of 2 tokens: 1 token pays for the push, 1 token is credited to the element for its future pop. Charge MULTIPOP an amortized cost of 0: each popped element spends the 1 token credited to it during its push. The credit balance is always non-negative (exactly the stack size). Total amortized cost: $2n$ for $n$ operations, giving $O(1)$ per operation.

</details>

</aside>

<aside class="exercise" data-difficulty="medium" data-type="symbolic" markdown="1">

**Exercise 5 (medium, symbolic).**

A dynamic table supports insertion and deletion. On insertion when full, it doubles capacity. On deletion when the load factor drops below 1/4, it halves capacity. Define a potential function and show that both operations have $O(1)$ amortized cost.

<details class="exercise__hint"><summary>Hint</summary>

Use $\Phi = 2 \cdot \text{size} - \text{capacity}$ when load factor $\ge 1/2$ and $\Phi = \text{capacity}/2 - \text{size}$ when load factor $< 1/2$.

</details>

<details class="exercise__answer"><summary>Answer</summary>

Define $\Phi = \begin{cases} 2s - c & \text{if } s \ge c/2 \\ c/2 - s & \text{if } s < c/2 \end{cases}$ where $s$ = size, $c$ = capacity. For insertion without resize: $\hat{c} = 1 + \Delta\Phi \le 3$. For insertion with resize (double): the new capacity is $2c$, old $s = c$, new $s = c + 1$; $\hat{c} = (c+1) + 2(c+1) - 2c - (2c - c) = 3$. For deletion without resize: $\hat{c} = 1 + \Delta\Phi \le 3$. For deletion with resize (halve): similar calculation gives $\hat{c} \le 3$. So both operations are $O(1)$ amortized.

</details>

</aside>

<aside class="exercise" data-difficulty="medium" data-type="short-answer" markdown="1">

**Exercise 6 (medium, short answer).**

Explain why the potential method is more general than the accounting method.

<details class="exercise__answer"><summary>Answer</summary>

The accounting method stores a credit balance that is a single non-negative number. The potential method allows $\Phi$ to be any non-negative function of the data structure state — it can depend on the distribution of elements, the tree shape, or any structural property. This makes the potential method applicable to data structures where the "bank balance" metaphor is too crude, such as splay trees (where the potential is a sum of logarithms of subtree sizes).

</details>

</aside>

<aside class="exercise" data-difficulty="hard" data-type="symbolic" markdown="1">

**Exercise 7 (hard, symbolic).**

A binary counter stores an integer in binary, incrementing by 1 at each step. The cost of an increment is the number of bits flipped. Show that $n$ increments starting from 0 have total cost $O(n)$ using the potential method.

<details class="exercise__hint"><summary>Hint</summary>

Define $\Phi$ as the number of 1-bits in the counter.

</details>

<details class="exercise__answer"><summary>Answer</summary>

Define $\Phi(D_i) = $ number of 1-bits in the counter after operation $i$. Starting from 0, $\Phi(D_0) = 0$ and $\Phi \ge 0$ always. An increment flips a suffix of the form $011\ldots1$ to $100\ldots0$: if there are $t$ trailing 1s, the cost is $t + 1$ (flip $t$ ones and one zero). The potential change: $\Phi(D_i) - \Phi(D_{i-1}) = (\text{new 1s}) - (\text{flipped 1s}) = 1 - t$. So $\hat{c}_i = (t+1) + (1-t) = 2$. The amortized cost of each increment is exactly 2. Total cost: $\sum \hat{c}_i = 2n$, so $O(n)$.

</details>

</aside>

<aside class="exercise" data-difficulty="hard" data-type="symbolic" markdown="1">

**Exercise 8 (hard, symbolic).**

Prove that a dynamic table that grows by a factor of 3 (tripling instead of doubling) also has $O(1)$ amortized insertion, but with a different constant.

<details class="exercise__answer"><summary>Answer</summary>

Define $\Phi(D_i) = 2 \cdot \text{size} - \text{capacity}$. For insertion without resize: $\hat{c} = 1 + 2 = 3$. For insertion with resize: old size $= k$, old capacity $= k$, new capacity $= 3k$, new size $= k + 1$. Actual cost $c_i = k + 1$ (copy $k$, write new). $\hat{c}_i = (k+1) + (2(k+1) - 3k) - (2k - k) = k + 1 + 2 - 3k + k = 3$. So the amortized cost is 3, matching the doubling case. More generally, for growth factor $\alpha > 1$, using $\Phi = 2s - c$ gives $\hat{c} = k + 1 + (2(k+1) - \alpha k) - k = k + 1 + 2 - \alpha k = 3 + k(2 - \alpha)$. For $\alpha \ge 2$, $\hat{c} \le 3$. For $1 < \alpha < 2$, $\hat{c}$ grows with $k$, so we need $\Phi = (\alpha/(\alpha - 1))s - c$ to get constant amortized cost.

</details>

</aside>

## Advanced results [Master]

The potential method is the most powerful and general framework for amortized analysis. Two advanced applications demonstrate its range.

**Theorem 1 (splay tree amortized analysis).** Every operation on a splay tree (access, insert, delete, join, split) takes $O(\log n)$ amortized time, where $n$ is the number of nodes [ref: TODO_REF Tarjan 1985]. The potential function is $\Phi = \sum_{x \in T} \log_2 s(x)$, where $s(x)$ is the size of the subtree rooted at $x$. The access lemma shows that splaying a node $x$ to the root changes the potential by at most $3(\log_2 s_{\text{new}}(x) - \log_2 s_{\text{old}}(x)) + 1 = O(\log n)$, which bounds the amortized cost. The static optimality theorem shows that splay trees perform within a constant factor of the optimal static binary search tree for any access sequence.

**Theorem 2 (union-find with path compression).** The union-find data structure with union by rank and path compression has amortized cost $O(\alpha(n))$ per operation, where $\alpha$ is the inverse Ackermann function [ref: TODO_REF Tarjan 1985]. The analysis uses a potential function based on the "level" and "index" of each node in the union-find forest. The amortized cost of a find operation is $O(\alpha(n))$ because each step along the find path either decreases the potential (a "good" step) or is one of at most $\alpha(n)$ "bad" steps per node. Since $\alpha(n) \le 4$ for all practical $n$, the amortized cost is effectively constant.

**Synthesis.** The potential method is the foundational reason amortized analysis extends beyond simple averaging to give tight bounds on sophisticated data structures: the potential $\Phi$ encodes the structural state of the data structure as a numerical quantity, and the amortized cost formula $\hat{c} = c + \Delta\Phi$ turns the analysis into an accounting exercise. The central insight is that the potential serves as a "savings account" built during cheap operations and spent during expensive ones, and putting these together with the right potential function for each data structure (subtree sizes for splay trees, levels for union-find, load factor for dynamic tables) yields tight $O(1)$ or $O(\log n)$ amortized bounds. This builds toward the analysis of network flow in [47.04.03] where augmenting paths are charged against a distance potential, and it appears again in [47.04.02] where red-black tree rebalancing is $O(1)$ amortized per update.

## Full proof set [Master]

**Proposition 1 (dynamic table doubling: $O(1)$ amortized).** *$n$ insertions into a dynamic table that doubles when full cost $O(n)$ total.*

*Proof.* The table doubles at sizes $1, 2, 4, \ldots, 2^k$ where $2^k$ is the smallest power of 2 at least $n$. The copy costs are $1 + 2 + 4 + \cdots + 2^{k-1} = 2^k - 1 < 2n$. The $n$ insertions each cost 1 to write. Total: $n + 2n - 1 = O(n)$. $\square$

**Proposition 2 (binary counter: $O(1)$ amortized per increment).** *$n$ increments of a binary counter starting from 0 flip $O(n)$ bits total.*

*Proof.* Use $\Phi = $ number of 1-bits. Each increment flips $t + 1$ bits (where $t$ is the number of trailing 1s). The potential decreases by $t - 1$: $t$ ones become zeros and one zero becomes a one. Amortized cost: $(t+1) + (1 - t) = 2$. Total: $2n$. $\square$

**Proposition 3 (multipop stack: $O(1)$ amortized).** *A stack supporting PUSH, POP, and MULTIPOP($k$) has $O(1)$ amortized cost per operation using the potential $\Phi = s$ (stack size).*

*Proof.* PUSH: actual cost 1, $\Delta\Phi = 1$, amortized $= 2$. POP: actual cost 1, $\Delta\Phi = -1$, amortized $= 0$. MULTIPOP($k$): actual cost $\min(k, s)$, $\Delta\Phi = -\min(k, s)$, amortized $= 0$. Since $\Phi \ge 0$ and $\Phi(D_0) = 0$, total amortized cost $\ge$ total actual cost. Total amortized for $n$ operations: at most $2n$. $\square$

## Historical & philosophical context [Master]

Amortized analysis was introduced by Robert Tarjan in his 1985 paper "Amortized Computational Complexity" [ref: TODO_REF Tarjan 1985]. Tarjan developed the potential-function framework to analyse splay trees, which he had introduced with Daniel Sleator in 1983. The key insight was that splay trees, despite having no explicit balance condition, achieve $O(\log n)$ amortized performance — a result invisible to worst-case analysis, since individual operations can trigger long chains of rotations.

The term "amortized" comes from finance, where an expense is spread over multiple periods. Tarjan's contribution was to make this intuition mathematically precise via the potential function, turning a financial metaphor into a proof technique. The accounting method was known informally before Tarjan's paper, but the potential method unified and generalised it, enabling the analysis of union-find, splay trees, and dynamic tables within a single framework.
