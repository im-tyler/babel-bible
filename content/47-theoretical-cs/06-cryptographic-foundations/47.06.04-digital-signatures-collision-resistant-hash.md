## Intuition [Beginner]

A handwritten signature on a document proves you approved it. Anyone can see the signature and compare it to a known sample. A digital signature does the same thing for electronic messages: you sign with a private key, and anyone verifies with your public key.

The key difference from a physical signature is that a digital signature cannot be forged. If you sign message $m$ with your private key $sk$, producing signature $\sigma$, then no one can produce a valid signature on a different message $m'$ without knowing $sk$. This is guaranteed not by the difficulty of imitating handwriting but by the computational hardness of inverting a one-way function.

Hash functions are the compression workhorse of cryptography. A hash function takes an input of any length and produces a fixed-length output, like a fingerprint. A collision-resistant hash function has the property that no one can find two different inputs with the same output. This seems simple, but it is remarkably useful: you can sign the hash of a message instead of the message itself, compressing arbitrarily long messages to a fixed size before signing.

Together, signatures and hash functions form the backbone of digital trust. Every time you visit a secure website, your browser verifies a digital signature on the site's certificate. Every time you install a software update, the operating system verifies a signature on the update. And every one of those signatures involves a hash function.

## Visual [Beginner]

```
Digital signature workflow:

  Signer (Alice):                    Verifier (Bob):
  Private key sk                     Public key vk
    
  Message m                          Message m
  Hash: h = H(m)                     
  Sign: s = Sign(sk, h)              
                                     Receive (m, s)
                                     Hash: h' = H(m)
                                     Verify: Verify(vk, h', s)?
                                     Yes -> authentic
                                     No  -> forged

Hash function collision resistance:

  Input space: 2^256 possible inputs (or more)
  Output space: 2^256 possible hashes
  
  Finding x, y with H(x) = H(y) by brute force:
    Birthday attack: ~2^128 trials needed
    (This is infeasible -- 2^128 ~ 10^38)
```

| Primitive | Input | Output | Security property |
|---|---|---|---|
| Hash function $H$ | Any-length message | Fixed-length digest | Collision resistance |
| Signature $\text{Sign}(sk, m)$ | Message + private key | Signature $\sigma$ | Unforgeability |
| Verification $\text{Verify}(vk, m, \sigma)$ | Message + public key + sig | Accept/Reject | Completeness |

## Worked example [Beginner]

We illustrate RSA-based digital signatures with toy parameters. Alice uses RSA with $p = 7, q = 13$, so $N = 91$ and $\phi(N) = 72$. She picks $e = 5$ (coprime to 72) and computes $d = 5^{-1} \bmod 72$. Since $5 \times 29 = 145 = 2 \times 72 + 1$, $d = 29$.

Public key: $(N, e) = (91, 5)$. Private key: $d = 29$.

**Signing.** Alice wants to sign the message $m = 42$. She computes the signature $\sigma = m^d \bmod N = 42^{29} \bmod 91$.

Using repeated squaring: $42^2 \equiv 35 \pmod{91}$. $42^4 \equiv 42$. $42^8 \equiv 35$. $42^{16} \equiv 42$.

Now $42^{29} = 42^{16} \times 42^8 \times 42^4 \times 42$. Stepwise: $42 \times 35 \equiv 14$. $14 \times 42 \equiv 42$. $42 \times 42 \equiv 35$. So $\sigma = 35$.

**Verification.** Bob computes $35^5 \bmod 91$. $35^2 \equiv 42$. $35^4 \equiv 35$. $35^5 \equiv 35 \times 35 = 1225 \equiv 42 = m$. Verified.

## Formal definition [Intermediate+]

**Definition (digital signature scheme).** A digital signature scheme is a triple of PPT algorithms $(KeyGen, Sign, Verify)$:

- $KeyGen(1^n) \to (vk, sk)$: outputs a verification key and signing key.
- $Sign(sk, m) \to \sigma$: outputs a signature on message $m$.
- $Verify(vk, m, \sigma) \to \{0, 1\}$: outputs accept or reject.

*Correctness:* $Verify(vk, m, Sign(sk, m)) = 1$ for all $m$.

**Definition (EUF-CMA security).** A signature scheme is *existentially unforgeable under chosen-message attack* if for every PPT adversary $A$:

$$\Pr[\text{Verify}(vk, m^*, \sigma^*) = 1 \wedge m^* \notin Q] \leq \text{negl}(n)$$

where $(vk, sk) \leftarrow KeyGen(1^n)$, $A$ has oracle access to $Sign(sk, \cdot)$ with query set $Q$, and $(m^*, \sigma^*)$ is $A$'s output.

**Definition (collision-resistant hash function).** A family of functions $\mathcal{H} = \{h_k : \{0,1\}^* \to \{0,1\}^n\}_{k \in \{0,1\}^n}$ is collision-resistant if for every PPT algorithm $A$:

$$\Pr_{k}[A(k) = (x, y) : x \neq y \wedge h_k(x) = h_k(y)] \leq \text{negl}(n).$$

**Definition (Merkle-Damgard construction).** Given a compression function $f : \{0,1\}^{2n} \to \{0,1\}^n$, the Merkle-Damgard hash of a message $m = m_1 \| m_2 \| \cdots \| m_l$ (with appropriate padding) is:

$$h_0 = IV, \quad h_i = f(h_{i-1}, m_i), \quad H(m) = h_l.$$

### Counterexamples to common slips

- *"Collision resistance means no two inputs have the same hash."* By the pigeonhole principle, collisions always exist (the input space is larger than the output space). Collision resistance means *finding* a collision is computationally infeasible, not that collisions do not exist.

- *"A signature proves who sent the message."* A signature proves that someone with the private key signed it. The connection to identity (who holds the private key) is established by a public-key infrastructure (PKI), not by the signature scheme itself.

- *"MD5 and SHA-1 are collision-resistant."* Both have been broken: Wang et al. (2004) found collisions in MD5, and Stevens et al. (2017) found collisions in SHA-1. SHA-256 and SHA-3 are the current recommendations.

### Key theorem with proof [Intermediate+]

**Theorem (RSA-FDH is EUF-CMA in the random oracle model).** *The RSA Full-Domain Hash signature schemeâ€”where $Sign(sk, m) = H(m)^d \bmod N$ and $Verify(vk, m, \sigma)$ checks $\sigma^e \bmod N = H(m)$â€”is EUF-CMA secure in the random oracle model assuming the RSA problem is hard.*

*Proof sketch.* Let $A$ be an EUF-CMA adversary that makes $q_H$ hash queries, $q_S$ signing queries, and forges with probability $\varepsilon$. We build an algorithm $B$ that solves the RSA problem: given $(N, e, y)$, find $x$ with $x^e \equiv y \pmod{N}$.

$B$ sets $vk = (N, e)$ and simulates the random oracle and signing oracle for $A$:

*Random oracle simulation.* $B$ maintains a table $H$ of query-response pairs. For each new query $m$: with probability $1 - q_S/q_H$, set $H(m) = y \cdot r^e \bmod N$ for random $r$ (an RSA challenge embedded in the hash). With probability $q_S/q_H$, set $H(m) = r^e \bmod N$ for random $r$ (a "signable" value). Record $(m, r, \text{type})$.

*Signing oracle simulation.* When $A$ asks for a signature on $m$: if $m$ was a hash query of "signable" type, return $r$ (since $r^e = H(m)$, so $Sign = H(m)^d = (r^e)^d = r$). If $m$ was not queried, handle it fresh as signable.

*Forgery extraction.* $A$ outputs $(m^*, \sigma^*)$ with $\sigma^{*e} = H(m^*) \bmod N$. If $m^*$ was a hash query where $B$ embedded the RSA challenge: $H(m^*) = y \cdot r^e$, so $\sigma^* = (y \cdot r^e)^d = y^d \cdot r$. Then $y^d = \sigma^* / r \bmod N$, and $B$ outputs $(\sigma^* \cdot r^{-1})^1 = y^d$ as the RSA solution.

The probability $m^*$ is the challenge-embedded query is at least $(1/q_H)(1 - q_S/q_H) \approx \varepsilon / q_H$, giving a non-negligible RSA solver. $\square$

**Bridge.** The RSA-FDH construction is the foundational reason hash-then-sign works: the hash function randomises the message before the RSA operation, preventing algebraic attacks on structured messages, and this builds toward the Merkle-Damgard construction that extends fixed-size compression to arbitrary-length hashing. The central insight is that the hash function serves dual purposesâ€”compressing the message to the RSA domain and randomising it to prevent exploitation of RSA's multiplicative structureâ€”and this is exactly the pattern that appears again in [47.06.02] where hashing randomises the plaintext before RSA encryption (OAEP padding). Putting these together with the one-way function theory of [47.06.01] shows that collision-resistant hash functions can be built from one-way functions (via Naor-Yung UOWHFs), and this is the bridge between the abstract OWF and the concrete hash-then-sign paradigm.

## Exercises [Intermediate+]

<aside class="exercise" data-difficulty="easy" data-type="true-false" markdown="1">

**Exercise 1 (easy, true-false).**

True or false: the Merkle-Damgard construction preserves collision resistance: if the compression function is collision-resistant, then so is the full hash function.

<details class="exercise__answer"><summary>Answer</summary>

**True.** This is the Merkle-Damgard strengthening theorem. If an adversary finds a collision $H(m) = H(m')$ for the full hash, then either the final compression function evaluations collide (giving a compression function collision) or some intermediate chaining values collide with different message blocks, which also gives a compression function collision. So any collision in the full hash yields a collision in the compression function.

</details>

</aside>

<aside class="exercise" data-difficulty="easy" data-type="symbolic" markdown="1">

**Exercise 2 (easy, symbolic).**

A hash function outputs $n = 256$ bits. By the birthday bound, how many random inputs must an adversary try to find a collision with probability approximately $1/2$?

<details class="exercise__answer"><summary>Answer</summary>

The birthday bound gives $O(2^{n/2}) = O(2^{128})$ trials. For $n = 256$, the adversary needs approximately $2^{128} \approx 3.4 \times 10^{38}$ hash evaluations. This is infeasible with current technology.

</details>

</aside>

<aside class="exercise" data-difficulty="medium" data-type="symbolic" markdown="1">

**Exercise 3 (medium, symbolic).**

Prove the birthday bound: if $H : \{0,1\}^* \to \{0,1\}^n$ is modelled as a random function, then after $q$ queries, the probability of finding a collision is approximately $q^2 / 2^n$.

<details class="exercise__answer"><summary>Answer</summary>

After $q$ queries $x_1, \ldots, x_q$, the probability of no collision is:

$$\Pr[\text{no collision}] = \prod_{i=1}^{q-1} \left(1 - \frac{i}{2^n}\right) \approx \prod_{i=1}^{q-1} e^{-i/2^n} = e^{-\sum_{i=1}^{q-1} i/2^n} = e^{-q(q-1)/2 \cdot 2^n}.$$

So $\Pr[\text{collision}] \approx 1 - e^{-q^2/(2 \cdot 2^n)} \approx q^2 / (2 \cdot 2^n)$ when $q^2 \ll 2^n$.

Setting this to $1/2$: $q^2 / (2 \cdot 2^n) = 1/2$, giving $q = \sqrt{2^n} = 2^{n/2}$. Rubric: full credit for the product-form probability and the exponential approximation.

</details>

</aside>

<aside class="exercise" data-difficulty="medium" data-type="symbolic" markdown="1">

**Exercise 4 (medium, symbolic).**

Prove that the Merkle-Damgard construction preserves collision resistance. Specifically, show that a collision in $H$ yields a collision in the compression function $f$.

<details class="exercise__answer"><summary>Answer</summary>

Suppose $H(m) = H(m')$ with $m \neq m'$. Let $m = m_1 \| \cdots \| m_l$ and $m' = m_1' \| \cdots \| m_{l'}'$. After length padding, both have lengths encoded in their final blocks.

Working backwards: $H(m) = f(h_{l-1}, m_l) = H(m') = f(h_{l'-1}', m_{l'}')$. If $(h_{l-1}, m_l) \neq (h_{l'-1}', m_{l'}')$, this is a collision in $f$. If they are equal, then $h_{l-1} = h_{l'-1}'$ and $m_l = m_{l'}'$. Since $m_l = m_{l'}'$ and the length padding ensures the final blocks encode the same length, we must have $l = l'$. Continue backwards: $f(h_{l-2}, m_{l-1}) = f(h_{l-2}', m_{l-1}')$. Again, if inputs differ, this is an $f$-collision. If equal, continue. This process must eventually find an $f$-collision because $m \neq m'$ means at least one message block differs. At the first position where blocks differ (from the front), the chaining values going in are equal (by the forward propagation from $IV$), so $f$ is evaluated on different inputs giving the same outputâ€”a collision. Rubric: full credit for the backwards-chaining argument.

</details>

</aside>

<aside class="exercise" data-difficulty="medium" data-type="short-answer" markdown="1">

**Exercise 5 (medium, short answer).**

Explain the difference between collision resistance and second-preimage resistance. Which is stronger?

<details class="exercise__answer"><summary>Answer</summary>

*Collision resistance:* no efficient algorithm can find *any* pair $(x, y)$ with $x \neq y$ and $H(x) = H(y)$. *Second-preimage resistance:* given a specific $x$, no efficient algorithm can find $y \neq x$ with $H(y) = H(x)$.

Collision resistance is stronger: a collision-finding algorithm can be used to break second-preimage resistance (pick a random $x$, find any collision, and output the other element). The converse does not hold in general: second-preimage resistance does not imply collision resistance, though for practical hash functions the gap is small.

</details>

</aside>

<aside class="exercise" data-difficulty="hard" data-type="symbolic" markdown="1">

**Exercise 6 (hard, symbolic).**

The Naor-Yung construction builds a universal one-way hash function (UOWHF) from any one-way permutation. A UOWHF is weaker than a collision-resistant hash: the adversary must specify their target input $x$ *before* seeing the hash key. Construct a UOWHF from a one-way permutation $f$.

<details class="exercise__answer"><summary>Answer</summary>

Given a one-way permutation $f : \{0,1\}^n \to \{0,1\}^n$ and the Goldreich-Levin hard-core predicate $h(x, r) = \langle x, r \rangle$:

Define $G_k : \{0,1\}^{2n} \to \{0,1\}^n$ by $G_k(x, r) = (f^{(1)}(x), h(x, r), h(f(x), r), \ldots, h(f^{(k-1)}(x), r))$ compressed to $n$ bits using pairwise-independent hashing.

For UOWHF security: the adversary commits to $x$ before seeing the key $k$. The key determines which bits of the PRG output to reveal. By the hard-core property, finding $y \neq x$ with $G_k(y) = G_k(x)$ requires inverting $f$ (since the hard-core bits for $y$ and $x$ would need to agree on all revealed positions, which happens with negligible probability unless $y$ is related to $x$ in a way that allows inversion).

More precisely: Naor and Yung's construction uses a pairwise-independent hash family to compress. The key selects a hash function $g$ from the family, and the UOWHF is $g \circ PRG$. The adversary, having committed to $x$, must find $y$ with $g(PRG(y)) = g(PRG(x))$. By pairwise independence, this requires $PRG(y) = PRG(x)$ (with high probability over $g$), which requires finding a collision in the PRG. But the PRG's pseudorandomness property implies collision resistance against adversaries who commit to one input first. Rubric: full credit for the PRG-based construction and the target-preimage security argument.

</details>

</aside>

<aside class="exercise" data-difficulty="hard" data-type="symbolic" markdown="1">

**Exercise 7 (hard, symbolic).**

Prove that EUF-CMA security of a signature scheme implies that the scheme is also secure against random-message attacks (where the adversary sees signatures on random messages, not chosen ones).

<details class="exercise__answer"><summary>Answer</summary>

Let $(KeyGen, Sign, Verify)$ be EUF-CMA secure. Suppose for contradiction it is not secure against random-message attacks: there exists a PPT adversary $A'$ that, given signatures on random messages $m_1, \ldots, m_q$ (each chosen uniformly), forges a signature on a new message $m^*$ with non-negligible probability $\varepsilon$.

We build an EUF-CMA adversary $A$: receive $vk$, submit random messages $m_1, \ldots, m_q$ to the signing oracle (these are uniformly random, matching the random-message attack model), receive signatures $\sigma_1, \ldots, \sigma_q$, and feed them to $A'$. If $A'$ outputs a valid forgery $(m^*, \sigma^*)$, $A$ outputs it.

The distribution of messages seen by $A'$ is identical to the random-message attack setting (both are uniform). $A$'s queries are all to the signing oracle, which is allowed in EUF-CMA. So $A$ forges with the same probability $\varepsilon$ as $A'$, contradicting EUF-CMA security.

This is a direct reduction: the random-message attack is a restricted version of the chosen-message attack (fewer adversarial capabilities), so EUF-CMA security is strictly stronger. Rubric: full credit for the reduction argument.

</details>

</aside>

<aside class="exercise" data-difficulty="hard" data-type="short-answer" markdown="1">

**Exercise 8 (hard, short answer).**

Explain why length-extension attacks are possible on the basic Merkle-Damgard construction and how SHA-3 (Keccak) prevents them.

<details class="exercise__answer"><summary>Answer</summary>

In the basic Merkle-Damgard construction, $H(m \| m')$ can be computed from $H(m)$ and the length of $m$, without knowing $m$: simply initialise the chaining value to $H(m)$ and continue hashing $m'$. This is a length-extension attack: given $H(m)$ and $|m|$, one can compute $H(m \| \text{padding} \| m')$ for any $m'$.

This is problematic for MAC constructions like $MAC_k(m) = H(k \| m)$: an attacker who sees $MAC_k(m)$ can compute $MAC_k(m \| \text{padding} \| m')$ without knowing $k$.

SHA-3 (Keccak) uses the sponge construction instead of Merkle-Damgard. The sponge absorbs the message in blocks and then squeezes output. The internal state after absorbing the message is not exposed as the hash output; instead, additional permutation rounds are applied before squeezing. This breaks the length-extension property because the output does not directly reveal the internal state that would be needed to continue hashing.

</details>

</aside>

## Advanced results [Master]

Three results extend the signature and hash function framework.

**Result 1 (Rompel's theorem: OWF implies signatures).** Rompel (1990) proved that one-way functions are necessary and sufficient for the existence of EUF-CMA secure signature schemes. The sufficiency direction constructs a UOWHF from any OWF (via the Naor-Yung technique), then builds a tree-based signature scheme where each node in a Merkle tree authenticates its children. The construction is polynomial-time but impractical due to large constants. Practical signature schemes (RSA, DSA, EdDSA) use specific number-theoretic assumptions for efficiency [ref: TODO_REF Rompel 1990].

**Result 2 (sponge construction and SHA-3).** The Keccak sponge construction (Bertoni, Daemen, Peeters and Van Assche, 2011), selected as SHA-3, uses a fixed-length permutation $f : \{0,1\}^b \to \{0,1\}^b$ (where $b = 1600$ for SHA-3). The message is absorbed into the state in $r$-bit blocks (rate $r$), with $c = b - r$ bits of capacity untouched. The output is squeezed from the state. The security proof shows that distinguishing the sponge output from random requires either inverting the permutation or finding a collision in the capacity part, both of which are assumed to be hard [ref: TODO_REF Arora-Barak Ch. 9].

**Result 3 (aggregate and multi-signatures).** Boneh, Gentry, Lynn and Shacham (2003) introduced aggregate signatures, where $n$ signatures on $n$ different messages from $n$ different signers can be combined into a single short signature. This is achieved using bilinear pairings on elliptic curves: the pairing equation $e(g^a, g^b) = e(g, g)^{ab}$ allows verification of multiple signatures simultaneously. Aggregate signatures are used in blockchain systems (certificates, block headers) to compress authentication data [ref: TODO_REF Arora-Barak Ch. 9-10].

**Synthesis.** Digital signatures and collision-resistant hash functions form the authentication layer of cryptography: signatures provide unforgeable attestation, and hash functions provide the compression and randomisation that make signatures efficient and secure. The central insight is that the hash-then-sign paradigm reduces the security of the signature scheme to two independent assumptions (collision resistance of the hash and security of the underlying primitive), and this builds toward Rompel's theorem showing that the entire construction chain rests on one-way functions. This is exactly the pattern that appears again in [47.06.01] where OWFs provide the foundation, and it generalises to the full cryptographic stack: OWFs yield hash functions, hash functions yield signatures, and signatures yield authentication and non-repudiation. Putting these together with the zero-knowledge proofs of [47.06.03] shows that the Fiat-Shamir heuristic converts ZK identification protocols into signatures, bridging the privacy and authentication layers of cryptography.

## Full proof set [Master]

**Proposition 1 (Merkle-Damgard preserves collision resistance).** *If $f : \{0,1\}^{2n} \to \{0,1\}^n$ is a collision-resistant compression function, then the Merkle-Damgard hash $H_f$ built from $f$ with proper length padding is collision-resistant.*

*Proof.* Suppose an adversary finds $m \neq m'$ with $H_f(m) = H_f(m')$. Let $m = m_1 \| \cdots \| m_l$ and $m' = m_1' \| \cdots \| m_{l'}'$ after padding (which encodes the original message length in the final block).

Since $m \neq m'$ and the length padding ensures different-length messages have different final blocks, there exists an index where the padded messages differ. Let $j$ be the largest index such that $(h_{j-1}, m_j) \neq (h_{j-1}', m_j')$.

We have $h_j = f(h_{j-1}, m_j)$ and $h_j' = f(h_{j-1}', m_j')$. Since $j$ is the largest index where inputs differ, for all $i > j$: $(h_{i-1}, m_i) = (h_{i-1}', m_i')$. So $h_j = h_j'$ (they propagate to the same final hash). Therefore $f(h_{j-1}, m_j) = f(h_{j-1}', m_j')$ with $(h_{j-1}, m_j) \neq (h_{j-1}', m_j')$.

This is a collision in $f$. So any collision in $H_f$ yields a collision in $f$, contradicting $f$'s collision resistance. $\square$

**Proposition 2 (RSA-FDH security reduction).** *If there exists a PPT adversary $A$ that breaks EUF-CMA security of RSA-FDH with probability $\varepsilon$ making $q_H$ hash queries, then there exists a PPT algorithm $B$ that solves the RSA problem with probability $\Omega(\varepsilon / q_H)$.*

*Proof.* $B$ receives $(N, e, y)$ and must find $x$ with $x^e = y \bmod N$. $B$ picks a random index $j^* \in \{1, \ldots, q_H\}$ and simulates the random oracle:

- For query $j \neq j^*$: pick random $r_j$ and set $H(m_j) = r_j^e \bmod N$.
- For query $j^*$: set $H(m_{j^*}) = y$.

When $A$ requests a signature on message $m$: if $m$ was queried at index $j \neq j^*$, return $r_j$ (since $r_j^e = H(m)$, so $r_j = H(m)^d$). If $m$ was queried at $j^*$, abort (cannot sign without knowing $y^d$). If $m$ was not queried, handle it as a new non-$j^*$ query.

$A$ outputs forgery $(m^*, \sigma^*)$. If $m^* = m_{j^*}$ (probability $1/q_H$), then $\sigma^{*e} = H(m^*) = y \bmod N$, so $\sigma^* = y^{1/e} = y^d$ is the RSA solution.

The probability $B$ succeeds is $(\varepsilon - q_S/q_H) / q_H \approx \varepsilon / q_H$ (subtracting the probability of aborting on a signing query for $m_{j^*}$). $\square$

## Historical & philosophical context [Master]

The concept of digital signatures was introduced by Diffie and Hellman (1976) in the same paper that proposed public-key cryptography. The first practical signature scheme was RSA (1978), where signing is encryption with the private key.

The formal security definition (EUF-CMA) was given by Goldwasser, Micali and Rivest (1988), who also introduced the distinction between existential and universal forgery. The Rompel theorem (1990) showing that OWFs suffice for signatures was a landmark, establishing that the entire authentication infrastructure rests on the same minimal assumption as symmetric cryptography.

The practical importance of hash functions was underscored by the break of MD5 (Wang et al. 2004) and SHA-1 (Stevens et al. 2017), which demonstrated that collision resistance is a property of specific constructions, not an abstract guarantee. The transition to SHA-256 and SHA-3 is ongoing across the software ecosystem.

The philosophical lesson is that authentication, like encryption, reduces to computational hardness: the impossibility of forging a signature is no more absolute than the impossibility of factoring a number, but it is sufficient for practical purposes [ref: TODO_REF Arora-Barak Ch. 9].
